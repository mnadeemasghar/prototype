1. Introduction

1.1 Purpose

This Software Requirements Specification (SRS) defines the functional and non-functional requirements for the Alsultania Travel CRM & Management System — a purpose-built, full-stack travel agency platform. It serves as the authoritative requirements baseline for design, development, testing, and client acceptance across all project phases.

1.2 Scope

The System consolidates the following into a single unified platform:

  • Lead capture, assignment, and lifecycle management
  • Customer profiles and booking workflows
  • Financial accounting, invoicing, and expense controls
  • Staff performance and HR metrics
  • WhatsApp automation and multi-channel notifications
  • Reporting, analytics, and multi-branch scalability

In scope: All modules defined in Phases 0–3 of the accepted proposal.

Out of scope (unless agreed in writing post-blueprint approval):

  • Custom branding/design beyond structural blueprint
  • Third-party WhatsApp API provisioning (Client responsibility)
  • Server/hosting provisioning (Client responsibility)
  • Features not mapped in the approved Phase 0 blueprint
  • Public website inquiry with SMS OTP (SRS §4.1.3b) unless accepted on Phase 0 Acceptance Form

2. Overall Description

2.1 Product Perspective

The System is a monolithic full-stack web application:

  • Backend: Laravel (PHP)
  • Database: MySQL
  • Frontend: Laravel Blade with component-based UI
  • Messaging: WhatsApp Business API (Client-provisioned)
  • Design baseline: Figma blueprint (structure, flows, fields — not branding)

2.2 Business Objectives

  1. Eliminate fragmented tools and manual lead entry
  2. Automate WhatsApp inquiry capture and agent follow-ups
  3. Centralise lead-to-booking lifecycle tracking
  4. Provide financial visibility and staff performance metrics
  5. Support multi-branch growth with segregated data and consolidated reporting

2.3 User Classes

Role Description
Administrator Full system access — users, roles, settings, branches, automation rules
Sales Manager Team oversight — lead assignment, approvals, reports, performance
Travel Agent Day-to-day CRM — leads, customers, bookings, WhatsApp conversations
Finance User Accounting, invoices, payments, expenses (Phase 2)
HR/Manager Staff performance, attendance, HR reports (Phase 2)

2.4 Operating Environment

  • Desktop-first web application (1440px design baseline)
  • Modern browsers (Chrome, Edge, Firefox, Safari — latest two versions)
  • Client-provisioned server with PHP 8.x, MySQL 8.x, HTTPS
  • WhatsApp Business API credentials provided by Client before Phase 1 go-live

2.5 Constraints

  • Development shall not commence until Phase 0 blueprint is client-approved in Figma
  • Approved blueprint is locked; scope changes require written agreement
  • 3-month delivery timeline contingent on timely client feedback and third-party readiness
  • WhatsApp API and server provisioning are Client dependencies

2.6 Assumptions

  • Client designates active email addresses for formal project communication
  • Agents have stable internet access during business hours
  • Lead sources include WhatsApp, website forms, walk-in, and referral
  • Currency for financial modules defaults to AED unless configured otherwise
